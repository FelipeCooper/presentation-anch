Backend

// what is plan mode? Plan mode

Cons

// spec-driven development SDD

What is Spec-Driven
Development?

A structured workflow where every feature goes through documented stages before code is written.

PRD
→ What are we building and why? Requirements, scope, constraints.
Tech Spec
→ How will we build it? Architecture, data models, API contracts.
Tasks
→ Granular breakdown. Each task is one executable unit.
Code
→ Execution with rules active. Structure, libs, patterns enforced.
Review
→ Agent reads PRD + spec + diff — grounded, not generic.
Each artifact is context for the next step. The model never starts from zero.
// .claude/rules/

Rules: your team's
standards, codified.

Rules are markdown files Claude reads automatically. They define what "right" looks like.

CLAUDE.md monolithic

Everything, always

React rules compete with DB rules during a migration. High priority everywhere = priority nowhere.

.claude/rules/*.md

Scoped by path

Frontend rules fire on frontend/**. Architecture rules fire globally. Right context, right moment.

# .claude/rules/stack.md
Backend: Go, Chi router, pgx, slog, golang-migrate
Frontend: React 19, TypeScript strict, Vite, TanStack Query
Styling: Tailwind CSS v4
Testing: Vitest + Playwright (E2E)
No: axios, any types, default exports, console.log
// .claude/rules/architecture.md

3 layers. Hard rules.
Domain knows nothing.

Dependency direction: app → domain ← foundation. Domain never imports the others.

foundation/

Infrastructure. DB, HTTP clients, CoinGecko API, config, logger. Implements interfaces domain defines.

domain/

Business logic + interfaces. Zero external imports. Defines what it needs, never how.

app/

Orchestration. Chi handlers, dependency injection in wire.go, CORS middleware.

// domain/portfolio/pricer.go — never knows CoinGecko exists
type AssetPricer interface {
   GetPrices(ctx context.Context, ids []string) (map[string]PriceData, error)
}

// .claude/commands/ Commands

Your workflow in
one keystroke.

Custom slash commands that wrap repeatable prompts. Stored in .claude/commands/, committed to git.

/create-prd

Reads a template, asks clarifying questions, generates a structured PRD with scope, constraints, and acceptance criteria.

/create-techspec

Reads the PRD as input, outputs a tech spec with data models, API contracts, and architecture decisions.

/create-tasks

Reads PRD + tech spec, generates a granular task list. Each task is a self-contained unit of work.

/run-task

Picks the next task, reads all context (PRD, spec, existing code), executes with rules enforced.

Every dev on the team runs /create-prd and gets the same structured output. No more "how did you prompt that?"
// .claude/skills/ Skills

Domain expertise
on demand.

Markdown files that teach Claude specific domain knowledge. Like a teammate reading the docs first.

golang-pro

Idiomatic Go patterns: error handling with %w, interfaces as contracts, table-driven tests, goroutine safety, slog logging.

react-best-practices

Functional components only, TanStack Query for server state, proper memo/callback usage, accessibility patterns.

# Skills vs Rules

Rules = always active (or path-scoped). Short. "Do this."
Skills = loaded on demand. Deep. "Here's how to think about this."

# Skills activate when Claude touches relevant code
# or when explicitly triggered by a command.
// .claude/agents/ Agents

Sub-agents with
a persona.

Agents are markdown-defined personas that review, test, or validate work with specific criteria.

task-reviewer agent

Reads PRD + spec + diff — feedback is grounded

Classifies issues as CRITICAL / MAJOR / MINOR. Checks against coding standards (naming, function size, test coverage). Generates [num]_task_review.md artifact for each task.

Without agents
"Review this code" → generic feedback about variable names and missing comments.
With task-reviewer
Checks domain boundary violations, missing error mapping, test coverage gaps against spec.
// model context protocol MCP

Plug into
real systems.

MCP servers give Claude access to external tools and data sources. Claude can read, query, and act on real data.

GitHub MCP

Create PRs, read issues, check CI status, comment on reviews — without leaving the terminal.

PostgreSQL MCP

Query your database schema, run read-only queries, validate migrations against live data.

Slack MCP

Post updates, read channel history, notify the team — integrated into your dev workflow.

Custom MCP

Build your own. Any API, any tool. Claude calls it like a native function.

MCP turns Claude from "generate code" into "act on your behalf" across your entire toolchain.
